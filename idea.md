## 🔥 ไอเดียใหม่ (พี่กันเสนอ 2026-07-18 ตอนต่อ API)

**C. 🤝 Ensemble — รันครั้งเดียว ให้ AI หลายเจ้าช่วยกัน แล้วรวมผล**
- พี่กันสั่ง 1 ครั้ง → เรียก AI 2+ เจ้าพร้อมกัน (เช่น Gemini + GPT-4.1) → เอาผลมา**รวมกัน** (union คำที่แยกได้ + dedupe คำซ้ำ)
- เหตุผล: แต่ละเจ้าจับคำงามไม่เหมือนกัน รวมกันได้คำครบกว่าเจ้าเดียว
- วิธี: `Promise.all([runAI(gemini), runAI(gpt)])` → merge items (dedupe by text, รวม source/kind, ถ้าขัดกันเลือกอันที่ดีกว่า) · อาจโชว์ว่าคำไหนมาจากเจ้าไหน (badge)
- ⚠️ ข้อคิด: จ่ายเงิน 2 เท่า (2 เจ้า) · ช้ากว่า (รอทั้งคู่) · ต้องออกแบบ UI เลือกหลายเจ้า + logic รวม

**D. 📝 Prompt แยกต่อ AI (แทน prompt เดียวใช้ทุกเจ้า)**
- ปัจจุบัน: prompt เดียว (`lib/prompt.js`) ใช้ทุกเจ้า · ปัญหา: ปรับเพื่อเจ้าหนึ่งอาจกระทบเจ้าอื่น
- ไอเดีย: ให้แต่ละเจ้ามี prompt override ของตัวเอง (ใน providers หรือแยกไฟล์) · เจ้าไหนไม่ตั้ง = ใช้ prompt กลาง
- พี่กันกลัวปรับ prompt แล้วเจ้าอื่นพัง — ระบบนี้แก้ปัญหานั้น (จูนแยกได้)

**E. 🎯 ระบบเทสความแม่นการจัดหมวด (benchmark)** — พี่กันสั่งเก็บไว้ 2026-07-18
- พี่กันสร้าง**ชุดคำทดสอบ + เฉลยหมวด** (คำนี้ควรอยู่หมวดไหน) → เว็บรันชุดเดิมด้วยทุกโมเดล → เทียบกับเฉลย → ออกมาเป็น **% ความแม่นการจัดหมวด**
- เห็นชัดว่าโมเดลไหนจัดหมวดแม่นสุด (มีตัวเลขวัด ไม่ใช่รู้สึกเอา)
- เหตุผล: พี่กันเป็นเภสัช ไม่เชี่ยวชาญภาษา → พึ่ง AI จัดหมวด → ยิ่งต้องรู้ว่าโมเดลไหนแม่นจริง
- ระหว่างนี้ทำ manual ได้ (จัดคำชุดเดียวหลายโมเดล → กด ⬇ ส่งออก → เทียบไฟล์เอง)

**F. 🐛 BUG: จัดคำใหม่ทับคำที่ค้างในตรวจทาน** (พี่กันเจอ 2026-07-18)
- อาการ: หน้าตรวจทานมีคำค้าง (ยังไม่กดบันทึกเข้าคลัง) → กดจัดคำใหม่จากหน้าเพิ่มคำ → **คำค้างเก่าหายหมด** ถูกแทนที่ด้วยชุดใหม่
- ต้นเหตุ: `process()` ใน `WordBankApp.jsx` → `setState({ review })` แทนที่ทั้งชุด ไม่ได้ append
- ควรแก้: จัดคำใหม่ให้ **ต่อท้าย** คำค้างเดิม (append + dedupe คำซ้ำ) แทนการทับ · ระวัง `reviewNovel` ถ้าคนละเรื่อง (จะรวมหรือแยก) · sync `wb_review` (replace → ต้องส่งชุดรวม) · อาจถามผู้ใช้ "เพิ่มต่อ หรือ เริ่มใหม่"

**G. 🔍 AI ตัวที่ 2 ตรวจซ้ำทั้งช่อ** (พี่กันสั่ง 2026-07-19)
- ปุ่มในช่อ → ส่งคำ + ประโยคต้นทางให้ AI อีกเจ้า คืน (1) คำที่ควรสกัดเพิ่ม (2) หมวด/กิ่งที่ควรเปลี่ยน
- 2 เจ้าเห็นต่าง → **ขึ้นป้ายให้ผู้ใช้เลือก ระบบห้ามเปลี่ยนเอง**
- ต้องมี: prompt ใบใหม่ (ตรวจงาน ≠ จัดคำ) · ขั้นเปรียบเทียบผล 2 ชุด · คอลัมน์ jsonb เก็บข้อเสนอ (ไม่งั้นรีเฟรชหาย) · ป้าย "2 เสียงไม่ตรงกัน"

**H. 💬 แชตในเว็บ** (พี่กันสั่ง 2026-07-19 · เลือกครบทุกความสามารถ)
- ถามตอบเรื่องคลัง · แนะนำคำตอนเขียน · สั่งจัดคำผ่านแชต · แก้ข้อมูลให้เลย (ต้องมีป๊อปยืนยันทุกครั้ง)
- **+ "เกลาหมวด"** — หมวดไหนคำเยอะเกิน ให้ AI เสนอแตกกิ่งย่อยใหม่
- บริบท: สรุปคลัง + รายชื่อกิ่งสด + ค้นเฉพาะคำที่เกี่ยวกับคำถาม (ไม่ส่ง 679 คำทุกครั้ง เปลืองและช้า)

## 1. โหมด "แคลร์จัดคำผ่านแชท" (ไม่ต้องใช้ API)

**แนวคิด:** แทนที่จะให้เว็บเรียก AI ผ่าน API (เสียเงิน/มีโควตา) ให้ **แคลร์ (Claude Code) เป็นคนอ่านคำเอง**
ทำตาม prompt เดียวกัน (แก้คำผิด / เก็บประโยคเต็ม / สกัดคำเด่น / แปะชนิด / จัดหมวด+หมวดย่อย)
แล้ว **insert เข้า Supabase โดยตรง** (ผ่าน Supabase MCP) → ให้ไปโผล่ที่ **หน้าตรวจทาน** (ไม่ใช่เข้าคลังเลย เผื่อพี่กันเช็คก่อน)

เสมือนว่า "หน้ากรอกหน้าแรก" กลายเป็นแชท Claude Code แบบที่กำลังใช้อยู่ตอนนี้

**ข้อดี:**
- ฟรี ไม่เสียค่า API เลย
- ใช้ Opus (แคลร์) = ฉลาดกว่า Gemini Flash
- ทำได้ทันทีตอนคุยกัน มี Supabase MCP อยู่แล้ว

**ข้อจำกัด:**
- ใช้ได้เฉพาะตอนพี่กันเปิด Claude Code คุยกับแคลร์ — เว็บที่ deploy แล้วคนอื่นเข้าใช้เองไม่ได้
- เป็นโหมด manual (ผ่านแชท) ไม่ใช่ระบบอัตโนมัติในเว็บ

**สรุป:** เป็น "โหมดเสริมส่วนตัว" คู่กับระบบ API (ไม่ใช่แทนกัน) — เหมาะกับ workflow ที่พี่กันคุยกับแคลร์อยู่แล้ว
ถ้าจะทำ: ต้องมีจุดกำหนดว่า "หน้าตรวจทาน" อ่านรายการรอตรวจจากไหน (เพิ่มตาราง wb_review หรือใช้ reviewed=false ใน wb_words)

## 3b. 📊 ประวัติการใช้ AI (เก็บไว้ ค่อยทำ — พี่กันสั่ง 2026-07-17)

เก็บ log ทุกครั้งที่เรียก AI จัดคำ:
- **เจ้าไหน** (provider) + **โมเดลอะไร** (model)
- **คำที่เข้ามา** (ข้อความ input ดิบ) · **คำที่ได้จากการตรวจทาน** (items ที่ AI แยก) · **คำที่บันทึกจริง** (หลังกดบันทึก)
- **token ที่ใช้** (input/output — ต้องอ่านจาก response ของแต่ละเจ้า) · **วันที่/เวลา** (timezone +7)
- แคลร์เสนอเพิ่ม: เวลาที่ใช้ประมวลผล (กี่วินาที) · จำนวนคำเข้า/ออก (30→40) · จำนวนที่ถูกข้ามเพราะซ้ำ · จำนวนแก้คำผิด · ผลสำเร็จ/error · ประมาณค่าใช้จ่าย (token × ราคาต่อเจ้า)
- ที่เก็บ: ตารางใหม่ `wb_ai_log` ใน Supabase · มีหน้าดูประวัติ (filter ตามเจ้า/วันที่) + สรุปยอดรวม token/ค่าใช้จ่าย
- ประโยชน์: รู้ว่าเจ้าไหนจัดดีสุด/คุ้มสุด · คุมค่าใช้จ่าย · ย้อนดูว่าคำไหนมาจาก batch ไหน
