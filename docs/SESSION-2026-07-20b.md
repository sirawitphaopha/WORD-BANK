## 🔴🔴 บทเรียนใหญ่ของรอบนี้ — git workflow (พี่กันอารมณ์เสียเพราะแคลร์ทำผิดกฎ)
สิ่งที่แคลร์ทำพลาด (ห้ามทำซ้ำ):
1. **ทำโค้ดเสร็จแล้ว commit/push ไปเลย ค่อยมาตามแก้ `CLAUDE.md`/`README` ทีหลัง** — ที่ถูกคือ**อัปเดตเอกสารก่อน commit** (เอกสารไปในคอมมิตเดียวกับโค้ด)
2. **push ลง branch ข้าง ๆ ก่อน (branch `claude/read-all-files-5lerkm` ที่คลาวด์ตั้งให้เริ่ม) แล้วค่อยเอาขึ้น main ทีหลัง** → บน GitHub เลยเห็น **main กับ branch แยกกัน** พี่กันต้องสั่ง "เอาขึ้น main" เอง · ที่ถูกคือ **เอาขึ้น main เท่านั้น ทีเดียวจบ**
3. **เผลอถือว่า "เอาขึ้น main" = อนุญาต push** ทั้งที่กฎบอกต้องมีคำว่า **"push / พุช" ตรง ๆ** เท่านั้น

### ✅ Flow งานที่ถูกต้อง (จดไว้บนสุด CLAUDE.md แล้ว — ต้องเดินตามทุกครั้ง)
1. พี่กันสั่งงาน
2. แคลร์สรุปก่อนลงมือ (ตอบตรง ๆ ก่อน)
3. ทำทีละขั้น · สงสัยถามก่อน ไม่เหมาเอง
4. เทส/พิสูจน์ว่าใช้ได้จริง + เอาให้พี่กันดู (พี่กันตัดสิน)
5. **อัปเดต `CLAUDE.md` + `README.md` ให้ตรงกับงานก่อน**
6. **เขียน commit** (โค้ด + เอกสาร รวมคอมมิตเดียว · จบด้วย `Co-Authored-By:`)
7. **รอพี่กันพิมพ์ "push" ตรง ๆ** — "ทำเลย / เอาขึ้น main" เฉย ๆ ไม่นับ
8. **push ขึ้น `main` เท่านั้น ทีเดียวจบ** (ห้ามทิ้งงานค้างบน branch อื่น · sync branch ให้ตรง main กันเห็นแยกกัน)
9. **push เสร็จ → เขียน `docs/SESSION-วันที่` → push ตามขึ้น main ทันที** (ไม่ต้องถาม · หลายรอบต่อวันเติม -b, -c)
