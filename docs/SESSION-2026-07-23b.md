## 🐛 งานหลักรอบนี้: บั๊กลบช่อ "ลบแล้วเด้งกลับ / ข้ามเครื่องไม่ซิงค์"

### อาการ (พี่กันรายงาน)
1. เมื่อวาน (22 ก.ค.) แคลร์แก้บั๊กลบช่อไปแล้วรอบหนึ่ง · พี่กันลบช่อออกจนหมด **โดยเทสจาก localhost**
2. วันนี้เปิด **เว็บจริง** (production · คนละเครื่อง คนละ localStorage) → **ช่อทั้งหมดที่ลบไปยังอยู่ครบ**
3. กดรีเฟรช → **ช่อทั้งหมดเด้งกลับมา** = แปลว่าลบไม่หายจริง ๆ
- พี่กันสั่ง: "ต้องทำระบบเรียลไทม์ และแก้บั๊กนี้ละ" + "เราต้องการสิ่งที่ดีที่สุด ไม่ต้องมาถาม"

### ต้นเหตุ (ลึกกว่าบั๊กเมื่อ 22 ก.ค.)
- **บั๊ก 22 ก.ค. แก้แค่ครึ่งเดียว:** เมื่อวานเปลี่ยนการลบจาก action `replace` (ลบทุกแถวเขียนใหม่ทั้งก้อน)
  → `remove` (targeted by ids) · แก้ได้แค่ **race ตอนลบรัว ๆ ในเบราว์เซอร์เดียว** (คำขอ replace payload
  ใหญ่ไปเสร็จทีหลัง เขียนช่อที่ลบแล้วกลับ)
- **ส่วนที่ยังไม่ได้แก้ = "ตาข่ายกันช่อหาย" ตอนเปิดแอป** (`componentDidMount` ใน `WordBankApp.jsx`)
  - ตาข่ายนี้ตั้งใจใส่ไว้กันช่อหายตอนปิดหน้ากลางคัน (คำค้างใน localStorage แต่ยังไม่ทัน sync ขึ้นคลาวด์)
  - มันเทียบ "ช่อในเครื่อง (localStorage) vs ช่อบนคลาวด์" · ช่อไหนมีในเครื่องแต่หายจากคลาวด์
    → เข้าใจว่า "ยังไม่ได้ sync" เลย `persistReviewNow()` **ดันกลับขึ้น Supabase**
  - **ปัญหา:** เครื่อง B ยังเก็บช่อเก่าใน localStorage (ที่เครื่อง A ลบไปแล้วบนคลาวด์)
    → ตาข่ายเครื่อง B เห็นว่า "มีในเครื่องแต่หายจากคลาวด์" → ดันช่อกลับขึ้นคลาวด์ = **ลบไม่มีวันหาย**
  - **แก่นปัญหา:** ระบบ **แยกไม่ออกว่า "ลบตั้งใจ" กับ "ยังไม่ได้ sync"** — ทั้งสองกรณีหน้าตาเหมือนกัน
    (ช่อมีในเครื่อง ไม่มีบนคลาวด์)

### วิธีแก้ 3 ชั้น (ทำครบแล้ว · build ผ่าน · ยังต้องเทสเว็บจริง)
1. **ตาราง tombstone `wb_review_deleted`** (`scripts/013_review_tombstone.sql`)
   - เก็บ **ชื่อช่อที่ "ตั้งใจลบ"** (batch = primary key + deleted_at) · เปิด RLS deny-all
   - 🔑 **พี่กันรันเองใน Supabase SQL Editor แล้ว** (เห็น "Success. No rows returned.")
     — MCP `execute_sql`/`apply_migration` ติด error -32003 (ต้อง approve ทุกครั้ง) สุดท้ายพี่กันรันเอง
   - ทุกการลบ (`remove`/`removeBatch`/`clear`) → ปักป้าย tombstone ผ่าน `tombstoneBatches()`
   - ตอน AI ใส่ช่อกลับ (`replace`) → เอาป้ายออกผ่าน `untombstoneBatches()` (ช่อนั้นไม่ใช่ช่อที่ลบแล้ว)
2. **ตาข่ายเคารพ tombstone** — เมธอดใหม่ `_mergeReview(cloudRaw, localRaw, deletedArr, fallbackNovel)`
   - ช่อที่อยู่ในรายการ tombstone (`deletedArr`) → **ห้ามฟื้นเด็ดขาด** ทั้งจาก local และ cloud
     + ลบออกจาก localStorage ด้วย
   - `/api/bootstrap` (GET) + `/api/review` (GET ใหม่) ส่ง `deletedBatches` กลับมาให้เว็บรู้ว่าห้ามฟื้นช่อไหน
3. **ซิงค์อัตโนมัติเกือบเรียลไทม์** — เมธอดใหม่ `syncReview()`
   - ยิง `GET /api/review` **ทุก 7 วินาที** (`setInterval`) + **ตอนสลับแท็บกลับมา** (visibilitychange/focus)
   - ลบช่อในแท็บ/เครื่องหนึ่ง → อีกที่เห็นตามเอง
   - กันชน: `_reviewSig()` ทำ signature ของ review เทียบก่อน `setState` (ถ้าเหมือนเดิม ไม่ setState)
   - หยุดซิงค์ถ้ากำลัง: เลือกช่ออยู่ / แก้คำอยู่ (`editCard`) / กำลังยิงงาน AI / มี op ค้าง (`_rvPending`)
     (กันซิงค์มาทับตอนผู้ใช้กำลังทำอะไรอยู่)

### 🔴 บทเรียน / ข้อควรระวังสำหรับ AI คนถัดไป
- **"ลบ" ในระบบ offline-first ต้องมี tombstone เสมอ** ไม่งั้น "ลบตั้งใจ" กับ "ยังไม่ sync" แยกไม่ออก
  → ตรงกับสกิลกลาง §6: "tombstone `_del:true` แทน delete จริง กันติ๊กออกแล้วเด้งกลับข้ามเครื่อง"
- **เทสบั๊กแบบนี้ต้องเทสข้ามเครื่อง/ข้าม localStorage** — เทสในเบราว์เซอร์เดียว (localhost) ไม่เจอ
  เพราะ localStorage เดียวกัน · **ต้องเทส 2 แท็บ / 2 เครื่อง / production จริง**
- MCP Supabase ในคลาวด์ติด approve ทุกครั้ง (-32003) → งาน SQL ให้พี่กันรันเองใน SQL Editor สะดวกกว่า
