# CLAUDE.md — คลังคำ (Word Bank)

คู่มือสำหรับ AI (Claude) ที่มาทำงานต่อในโปรเจกต์นี้ — **อ่านให้จบก่อนลงมือทุกครั้ง**
ใช้ได้ทั้งในเครื่องและบนคลาว (cloud agent)

> 🚨🚨🚨 **กฎเหล็ก push/บันทึก — อ่านก่อนทุกครั้ง ห้ามลืม (เคยพลาดจริง 20 ก.ค. 2569)**
> **ลำดับทุกครั้งก่อนขึ้น main (ห้ามสลับ):** ① **แก้/อัปเดต `CLAUDE.md` + `README.md`** ให้ตรงกับงานที่ทำก่อน → ② **เขียน commit** (รวมโค้ด + เอกสารในคอมมิตเดียวกัน) → ③ พอพี่กันพิมพ์ "push" ค่อย **push ขึ้น main เท่านั้น ทีเดียวจบ** → ④ **เขียน `docs/SESSION-วันที่`** แล้ว push ตามขึ้น main · 🔴 **ห้ามทำโค้ดเสร็จแล้ว commit/push เลยโดยไม่อัปเดตเอกสารก่อน** (พลาดจริง 20 ก.ค. — ทำเสร็จ push ไปแล้วค่อยมาตามแก้ CLAUDE.md/README ทีหลัง)
> 1. **ห้าม push จนกว่าพี่กันพิมพ์คำว่า "push / พุช" ตรง ๆ** · คำว่า "ทำต่อ · ทำเลย · เอาขึ้น · เอาขึ้น main" ที่ไม่มีคำว่า push = **ยังไม่อนุญาต** (ถ้าไม่แน่ใจ = ถามก่อน)
> 2. 🎯 **เอาขึ้น `main` เท่านั้น** (พี่กันเป็น solo dev เจ้าของ repo) — งานสุดท้ายต้องอยู่บน **main** เสมอ · **ห้าม push ค้างไว้บน branch อื่นเด็ดขาด** (เช่น branch ที่ cloud ตั้งให้เริ่ม) เพราะจะทำให้ main กับ branch **แยกกันบน GitHub** · ถ้าคลาวด์บังคับให้เริ่มบน branch อื่นจริง ๆ พอพี่กันสั่ง push ให้ **รวบขึ้น main ทีเดียวให้จบ** ไม่ทิ้งไว้แยก · **อย่า push ลง branch ข้าง ๆ ก่อนแล้วค่อยย้ายมา main ทีหลัง**
> 3. **push ทีเดียวให้จบหลังพี่กันสั่ง** — อย่า push ทีละนิดก่อนได้รับอนุญาต หรือ push ไป branch ก่อนแล้วค่อยย้ายไป main ทีหลัง (ทำให้ประวัติแตกเป็นเสี่ยง ๆ บน GitHub — นี่คือสิ่งที่พลาดไป 20 ก.ค.)
> 4. **push เสร็จ = เขียน `docs/SESSION-YYYY-MM-DD.md` ทันที แล้ว push ตามขึ้น main ด้วย** (ไม่ต้องถาม) · หลายรอบในวันเดียวเติม `-b`,`-c`
> 5. **ก่อน push:** เช็ค `.gitignore` กัน `.env.local` หลุด · redact กุญแจ/อีเมล/โทเคนทุกชนิด
> 📖 รายละเอียดเต็มอยู่หัวข้อ **"push / เวอร์ชัน"** ด้านล่าง — กฎนี้ยกมาไว้บนสุดเพราะ **ต้องอ่านทุกครั้งก่อนแตะ git**
> 🤖 **มีระบบเด้งเตือนอัตโนมัติ (hook) แล้ว** (เพิ่ม 20 ก.ค. 2569) ที่โฟลเดอร์ `.claude/hooks/` + ตั้งค่าใน `.claude/settings.json` — ทำงาน 2 จังหวะ: (1) **ตอนเปิด session ทุกครั้ง** เด้งกฎ + สั่งอ่านสกิลเข้า context อัตโนมัติ (`session-start.sh`) (2) **ก่อน `git commit`/`git push` ทุกครั้ง** เด้งเช็คลิสต์ (`pre-git.sh` · กรองด้วย `if: Bash(git *)`) · **แก้เนื้อกฎที่เด้งได้จุดเดียว** ที่ `.claude/hooks/rules-core.txt` · เหตุผล: เขียนกฎไว้เฉย ๆ ไม่พอ AI เคยเผลอไม่อ่าน → ให้ระบบบังคับเด้งทุกครั้ง

> 🔬 **ก่อนตอบเรื่อง AI ทุกครั้ง — เปิดอ่าน `docs/AI-MODEL-TEST.md` ให้จบก่อนเสมอ**
> จังหวะที่ต้องอ่าน: พี่กันจะเทส AI · ถามว่าใช้รุ่นไหนดี · ให้วิเคราะห์ผลช่อ · จะแก้ `lib/prompt.js` · เจอ AI จัดคำแปลก ๆ
> ในนั้นมี **สรุปสั้นอยู่หัวไฟล์** (อ่าน 1 นาทีจบ) แล้วค่อยลงรายละเอียด — **ห้ามตอบเรื่องพวกนี้จากความจำ** เพราะเคยสรุปผิดมาแล้วหลายรอบ

### พี่กัน — เจ้าของโปรเจกต์ / ผู้ใช้
- เภสัชกรโรงพยาบาล **ไม่มีพื้นฐานการเขียน/อ่านโค้ดเลย** แต่เข้าใจ logic (ตรรกะการทำงาน) ของระบบได้ดีมาก
- งานอดิเรก: อ่านและเขียนนิยาย — โปรเจกต์นี้ทำไว้ใช้เก็บคำสวยๆ ที่เจอตอนอ่าน
- ทำเว็บคนเดียว จ่ายเอง เรียนรู้ไปเรื่อยๆ
- **อย่าคาดเดาว่าพี่กันรู้ศัพท์เทคนิค** (เช่น RLS, API, env, branch, deploy) — อธิบายด้วยภาษาคนหรืออนาล็อกง่ายๆ

## 📏 กฎการทำงาน (ทำตามทุกครั้ง — สำคัญมาก)

### ภาษา
- **ภาษาไทยเป็นหลัก** คำไหนแปลไทยได้ให้ใช้ไทย · ศัพท์เทคนิคที่ไม่มีคำไทยให้วงเล็บแปลกำกับ
- คุยกับพี่กัน = ภาษาคน (อธิบาย logic สำคัญด้วยเสมอ เผื่อพี่กันอยากเข้าใจว่ามันทำงานยังไง)
- 🚨🚨 **ห้ามพูดรหัสหมวด `c0`–`c8` กับพี่กันเด็ดขาด** — เรียกเป็น **"หมวด + เลข/ชื่อไทย"** เท่านั้น (เช่น "หมวด 8 บุคลิกภาพและลักษณะนิสัย" ไม่ใช่ "c7")
  - เหตุผล: รหัส c เป็นชื่อหลังบ้านล้วนๆ พี่กันไม่เคยเห็นในเว็บ (เว็บโชว์ "หมวด 1–9" + ชื่อไทยเสมอ) · แถมรหัส **เริ่มนับจาก 0** เลยเลื่อนจากเลขหมวดไป 1 ช่อง → เอามาพูดทีไรพี่กันสับสนทุกที
  - 🔴 เคยทำพลาดจริง (19 ก.ค. 2569): แคลร์บอกว่า "ย้ายคำบุคลิกไป c7" พี่กันอ่านเป็น "หมวด 7 (คำทับศัพท์)" เลยเข้าใจผิดว่าย้ายผิดหมวด ทั้งที่ c7 = หมวด 8 (บุคลิกภาพ) ถูกต้องอยู่แล้ว → วุ่นวายหลายรอบเพราะพูดรหัส
  - 🔑 ตารางแปลงไว้ใช้ในหัวเงียบๆ (ห้ามพ่นออกมา): c0=หมวด 1 · c1=หมวด 2 · c2=หมวด 3 · c3=หมวด 4 · c4=หมวด 5 · c5=หมวด 6 · c6=หมวด 7 (คำทับศัพท์) · c7=หมวด 8 (บุคลิกภาพ) · c8=หมวด 9 (อื่น ๆ)
- 🚨 **ข้อความใน UI (ปุ่ม/หัวข้อ/label/popup) ห้ามมีเครื่องหมาย `?`** เช่น ❌"ลบ?" → ✅"ยืนยันลบ"
- 🚨 **ข้อความใน UI ต้องเป็นภาษาทางการ ห้ามมี ค่ะ/นะคะ**
- ห้ามใช้ browser alert/confirm — ใช้ popup สวยๆ ในแอป หรือ inline error / toast แทน

### วิธีทำงาน
- **ตอบตรงๆ ก่อนเสมอ** ประโยคแรก = คำตอบ · **ก่อนลงมือสรุปให้พี่กันเข้าใจก่อน** ว่าจะทำอะไร เห็นผลอะไร
- **ทำทีละขั้น** รอพี่กันบอก "โอเค" ก่อนไปขั้นถัดไป
- 🚨 **สงสัย/ไม่แน่ใจ = ถามก่อน ห้ามเหมาเอง** — เห็นอะไรดูแปลกอย่าเพิ่งคิดว่าบั๊กแล้วแก้ อาจเป็นที่ตั้งใจ
- **ห้ามลบเนื้อหา/ฟีเจอร์โดยไม่ถามก่อน**
- **แก้บั๊กให้ส่ง/แก้ไฟล์เต็ม** ไม่แก้แค่บางส่วน
- **การกระทำย้อนยาก (ลบ/ยกเลิก) ต้องมี popup ยืนยันก่อนเสมอ**
- **อะไรที่คล้ายกันทำให้เหมือนกัน** รวมเป็น component กลางตัวเดียว (ปุ่ม/badge/popup ยืนยัน)

### push / เวอร์ชัน
- 🚨🚨 **ห้าม push โดยไม่ได้รับอนุญาต** — push ได้เฉพาะเมื่อพี่กันพิมพ์คำว่า **"push / พุช" ตรงๆ** เท่านั้น
  (คำว่า "ทำต่อ/ทำเลย/เอาขึ้นเมน" ไม่ใช่การอนุญาต push)
- push ขึ้น branch **main** โดยตรง (solo dev เจ้าของ repo)
  - 🔴 **เคยพลาดจริง 20 ก.ค. 2569:** cloud agent เริ่มบน branch `claude/read-all-files-5lerkm` แล้วแคลร์ push ลง branch นั้นก่อน (ตามที่ระบบคลาวด์สั่ง) → บน GitHub เลย **เห็น main กับ branch แยกกัน** พี่กันต้องสั่ง "เอาขึ้น main" เองอีกที · **ที่ถูก: พอพี่กันสั่ง push ให้รวบขึ้น main ทีเดียวให้จบ ไม่ push ลง branch ข้าง ๆ ก่อน** (ถ้าคลาวด์บังคับ branch จริง ๆ อย่างน้อยตอน push สุดท้ายต้องเอาเข้า main ให้ตรงกัน ไม่ทิ้งไว้แยก)
- **ก่อน push เช็ค .gitignore** กัน `.env.local` (มี service key ลับ) หลุด · redact secret ทุกครั้ง
- commit message ละเอียด เขียนสิ่งที่เปลี่ยน/เพิ่ม/แก้ · ไทย+อังกฤษ · จบด้วย `Co-Authored-By:`
- 🚨🚨 **push เสร็จ = เขียน `docs/SESSION-YYYY-MM-DD.md` แล้ว push ตามขึ้นไปด้วยทันที** (ไม่ต้องถาม)
  - หลายรอบในวันเดียว → เติมตัวอักษรท้าย `SESSION-2026-07-19b.md`, `-c`, `-d` …
  - เขียนละเอียดแบบ commit: ทำอะไร แก้บั๊กอะไร (ต้นเหตุ + วิธีแก้ + วิธีพิสูจน์) บทเรียนที่ห้ามซ้ำ สิ่งที่ค้างไว้
  - 🎯 **เหตุผลที่ต้องอยู่ใน repo ไม่ใช่แค่ในเครื่อง**: เวลาพี่กันสั่งงาน Claude บน**คลาวด์** (cloud agent) ตัวนั้นเห็นแต่ repo ไม่เห็นโฟลเดอร์ความจำในเครื่อง → ไฟล์ docs คือสิ่งเดียวที่ทำให้มันรู้จักพี่กัน รู้ว่าทำอะไรไปแล้ว และทำงานต่อได้ทันที
  - **เขียนให้ AI ที่ไม่เคยคุยกับพี่กันมาก่อนอ่านรู้เรื่อง** — บริบทการทำงาน ชื่อเรียก สไตล์ที่พี่กันชอบ ข้อที่เคยถูกท้วง ให้คงไว้ทั้งหมด
  - 🔑 **เอาออกเฉพาะของลับ**: อีเมล · กุญแจ/โทเคนทุกชนิด (`sb_secret_`, `eyJ`, `sk-`, `re_`, `AIza`) · รหัสผ่าน — นอกนั้นเก็บไว้ให้ครบ
  - ✍️ **เกณฑ์การเขียน: เขียนแบบที่ "แคลร์เองเปิดมาอ่านแล้วทำงานต่อได้ทันที"** — สมมติว่าไม่มีบริบทอะไรเลย ไม่เคยคุยกับพี่กันมาก่อน อ่านไฟล์นี้ไฟล์เดียวต้องรู้ว่า: ทำอะไรไปแล้ว ทำไมถึงทำแบบนั้น เคยพลาดอะไร ห้ามทำอะไรซ้ำ และเหลืออะไรต้องทำต่อ
  - ห้ามใช้ตัวย่อที่ตัวเองเท่านั้นเข้าใจ · ชื่อฟังก์ชัน/ไฟล์/คอลัมน์เขียนให้ครบ · บั๊กต้องมีครบ 3 อย่าง (อาการ + ต้นเหตุ + วิธีแก้/วิธีพิสูจน์)
  - เขียนฉบับเดียวใช้ได้ทั้ง 2 ที่ (ความจำในเครื่อง + docs ใน repo) ไม่ต้องเขียนย่อกับเต็มแยกกัน
  - 🔁 **ถ้าพบว่ามีวันไหนยังไม่มีไฟล์ใน  ให้เขียนย้อนหลังทันที** โดยดึงเนื้อหาจากบันทึกในโฟลเดอร์ความจำ (ห้ามแต่งขึ้นเอง) แล้วเรียบเรียงให้ AI ที่ไม่เคยคุยกับพี่กันอ่านรู้เรื่อง
  - ทุกวันต้องมีไฟล์ครบ ไม่ข้ามวัน · ก่อนจบงานแต่ละวันเช็คว่า  มีไฟล์ของวันนั้นแล้ว

- `wb_novels` (id, title unique)
- `wb_words` (id, text, original_text, meaning, category_id, **kind**, **subcategory**, **subpath**, **subpaths**(jsonb หลายกิ่ง), **highlight**, novel, reviewed, created_at)
  - `kind` = ชนิดคำ `'word'` | `'phrase'` | `'sentence'` (AI แปะให้ · กดเปลี่ยนได้หน้าตรวจทาน · ใช้กรองหน้าคลังคำ) — เพิ่มด้วย `scripts/004_add_kind.sql`
  - `subpath` = เส้นทางหมวดย่อยเต็ม (ลึกได้ 3 ชั้น) คั่นด้วย " / " — หน้าคลังคำสร้างต้นไม้จากค่านี้
  - `highlight` = คำสำคัญที่เดิมอยู่ในวงเล็บ (เก็บไว้เผื่อทำระบบไฮไลต์คำเน้น ยังไม่ได้ใช้)
- `wb_review` (ห้องพักคำตรวจทานบนคลาวด์) — id(=r_xxx ฝั่งเว็บ), text, original, meaning, kind, category_id, subpath, **source**(ประโยคต้นทาง), proposed_new, notes(jsonb), novel(**รายคำ** แต่ละช่ออาจคนละเรื่อง), position, **batch / batch_no / batch_at / batch_ai** (ช่อคำ · `scripts/007`) · กดบันทึก = ย้าย**เฉพาะช่อที่เปิด**ไป wb_words ช่ออื่นค้างต่อ · `scripts/005`
- `wb_ai_log` (ประวัติการใช้ AI / log ช่อ) — provider, provider_label, model, novel, input_text, input_chars, item_count, extracted_count, items_json, **saved_count/skipped_count**(เติมตอนบันทึก), tokens_in/out, **cost_usd**, duration_ms, status, error · `scripts/006` · **+ `batch_code`,`batch_no`,`typed_count`,`restored_count`,`new_branch_count`** (ผูกช่อ+สถิติถาวร · `scripts/012` · เติมผ่าน action `batchStats` ฝั่งเว็บ)
- `wb_branches` (ทะเบียนชื่อกิ่งหมวดย่อยทั้งหมด · `scripts/009`) — category_id, path, **source** (seed=โครงตั้งต้น · user=ผู้ใช้สร้าง · ai=AI เสนอแล้วรับ), created_at · unique(category_id,path)
